# Refactoring Rules

Этот документ фиксирует единые правила дальнейшего рефакторинга `naim-node`.
Цель правил: делать код более очевидным, логичным, объектным и устойчивым к дальнейшему развитию без изменения внешнего поведения системы.

## 1. Общие принципы

- Рефакторинг ведётся в сторону строгого объектного подхода.
- Поведение должно жить в классах, а не в больших наборах свободных функций.
- Каждый класс должен иметь одну ясную ответственность.
- Каждый файл должен рассказывать одну понятную историю.
- Внешнее поведение сохраняется:
  - CLI flags
  - HTTP endpoints
  - DB schema
  - desired-state schema

## 2. Entry Point и `main.cpp`

- `main.cpp` должен быть тонким.
- В `main.cpp` допускается только:
  - создание верхнеуровневого объектного entrypoint
  - вызов `Run()`
  - возврат exit code
- Логика orchestration, bootstrap, wiring и domain behavior в `main.cpp` не допускается.
- Глобальные `Run*App(...)` wrappers не используются.

## 3. Правило `App`

- Пустые прокладки запрещены.
- Отдельный `*App` создаётся только если у него есть собственная ответственность.
- Если `CompositionRoot` уже является фактической точкой входа, его можно называть `*App` и вызывать напрямую.
- Нельзя держать несколько слоёв только ради формы, если промежуточный слой ничего не добавляет.

## 4. Composition Root

- Object graph должен собираться в composition root.
- Composition root владеет long-lived зависимостями.
- Composition root не должен смешиваться с domain behavior.
- Если wiring становится слишком большим, его нужно делить на более узкие composition slices.

## 5. Один класс — один файл

- Это жёсткое правило для всех классов без исключений.
- Каждый класс обязан иметь:
  - отдельный `.h`
  - отдельный `.cpp`
- Правило распространяется на:
  - service classes
  - helper classes
  - policy classes
  - validator classes
  - support classes
  - adapters
  - repositories
  - interfaces
- Локальные классы внутри `.cpp` запрещены.

## 6. Helper classes

- Если helper стабилен и имеет осмысленную роль, он всегда выносится в отдельный файл.
- Tiny private helper class внутри `.cpp` не допускается.
- Если сущность достаточно оформлена, чтобы быть классом, она обязана жить в своём `.h/.cpp`.

## 7. Свободные функции

- Свободные функции нужно избегать максимально.
- Допустимый кандидат по умолчанию только один:
  - `main`
- Всё остальное по возможности переносится в методы классов.
- Если логика имеет владельца, она не должна существовать как свободная функция.

## 8. Структура папок

- Используется mirrored layout:
  - `include/...`
  - `src/...`
- Папки называются по смыслу и предметной области.
- Типовые подпапки:
  - `app`
  - `http`
  - `auth`
  - `bundle`
  - `plane`
  - `scheduler`
  - `read_model`
  - `infra`
  - `backend`
  - `config`
  - `install`
  - `run`
  - `platform`
  - `observation`
  - `state_apply`
- В `include` хранятся только `.h`.
- В `src` хранятся только `.cpp`.

## 9. Include Policy

- Include должен максимально отражать физическую структуру файла.
- Это фиксированный стандарт для всего репозитория.
- После перемещения файлов обязательно синхронизируются:
  - `CMakeLists.txt`
  - `.vscode/c_cpp_properties.json`
  - при необходимости `.vscode/settings.json`

## 10. Зависимости и API

- Большие `Deps`-мешки и публичные `std::function`-наборы запрещены.
- Вместо них используются:
  - явные named dependencies
  - интерфейсы там, где реально есть вариативность поведения
- Конструктор класса должен явно показывать, от чего зависит объект.
- Public API должен быть понятен по именам типов, а не по callback-подписям.

## 11. Ownership

- Ownership должен быть очевиден из кода.
- Owned зависимости по умолчанию выражаются через `std::unique_ptr`.
- Обязательные не-owning зависимости передаются по ссылке.
- `std::shared_ptr` используется только при реальном shared lifecycle/state.
- Скрытые глобальные mutable состояния не допускаются.

## 12. Интерфейсы и полиморфизм

- Интерфейсы вводятся там, где реально меняется поведение:
  - backend selection
  - local vs remote communication
  - transport
  - process execution
  - service management
  - telemetry collection
  - runtime engine
- DTO и value types не превращаются в искусственные иерархии.
- Полиморфизм используется осмысленно, а не ради формы.

## 13. Domain Services

- HTTP router отвечает только за orchestration.
- HTTP layer не должен владеть payload-building и domain logic.
- Domain logic не должна жить в `main.cpp`, bootstrap-файлах и случайных helper islands.
- Service должен быть owner’ом своей ответственности, а не прокладкой.

## 14. Persistence и `common`

- Shared-модули режутся по доменным границам.
- Giant files не допускаются как постоянное состояние.
- `sqlite_store` должен становиться facade над repository classes.
- JSON/import/runtime/planning logic должна жить в:
  - codec classes
  - builder classes
  - importer classes
  - evaluator classes

## 15. Runtime Components

- Для runtime-компонентов предпочтительны отдельные классы по ролям:
  - config loader
  - signal or stop service
  - status service
  - model resolver
  - engine host
  - router or server
- Лишние object layers, не несущие собственной ответственности, удаляются.

## 16. Отключённый код

- Общее правило: отключённый и неактуальный код удаляется.
- Неактуальный код не хранится в активной кодовой базе только как архив.
- Единственное разрешённое исключение:
  - `llama` в `worker`
- Для `llama` разрешено оставить код закомментированным и исключённым из object graph и сборки.
- Для всего остального действует правило удаления.

## 17. Проверка после каждого среза

- После каждого рефакторинга обязательно:
  - сборка целевого таргета
  - минимальный smoke test
- Если менялась структура include’ов или layout:
  - дополнительно проверяются IntelliSense fallback paths

## 18. Практический чеклист

Перед завершением каждого рефакторинга нужно проверить:

- Определён owner каждой ответственности.
- Удалены пустые прокладки.
- Каждый класс вынесен в отдельный `.h/.cpp`.
- Свободные функции устранены, кроме `main`.
- Callback-bag заменён на named dependencies или интерфейсы.
- Include-paths и структура папок синхронизированы.
- Целевой таргет собирается.
- Минимальный smoke test проходит.
