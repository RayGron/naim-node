## Findings

### 1. `comet-inferctl` link regression was fixed

Severity: high during discovery, fixed in follow-up

Initial failure:

- target: `comet-inferctl`
- failure point: final link step
- error family: `undefined reference` to `comet::runtime::ModelAdapter::*`

Root cause:

- `common/src/runtime/model_adapter.cpp` was added to `comet-common`
- `comet-inferctl` started using `ModelAdapter` from `infer_app.cpp` and
  `llama_rpc_runtime.cpp`
- `comet-inferctl` did not link `comet-common`

Fix applied:

- updated `comet-inferctl` to link `comet-common`

### 2. `hpc1` Git SSH transport is still not usable for `origin`

Severity: medium

Observed behavior:

- `git fetch --all --prune` on `hpc1` failed with `git@github.com: Permission denied (publickey)`

Impact:

- routine `pull` from the configured remote is not reliable on the host
- deployment flow currently depends on an HTTPS override rather than the saved remote config

Status:

- not a code regression in this slice
- still an operational issue worth fixing on the host

### 3. `comet-inferctl` asset lookup is cwd-sensitive

Severity: low

Observed behavior:

- running `./comet-inferctl list-profiles` from `build/linux/x64` failed with:
  - `runtime profiles not found: runtime/infer/runtime-profiles.json`
- running `build/linux/x64/comet-inferctl list-profiles` from the repo root succeeded

Impact:

- ad-hoc CLI use depends on the current working directory unless the caller uses the expected repo
  root layout

Status:

- not blocking for the current validation
- worth keeping in mind for operator tooling and local debugging UX
